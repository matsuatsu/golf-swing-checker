# 技術スタック

## 方針

Golf Swing Checkerは、教材を中心とした軽量な静的アプリとして作る。

最初の段階では、サーバー、データベース、ユーザーアカウントは持たせない。

教材ページを読みやすく表示し、ステップ完了チェックやチェックポイント内の計測値目安など、必要な部分だけをインタラクティブにする。

## 基本構成

- Docusaurus
- React
- TypeScript
- Markdown / MDX
- Node.js v24
- Corepack
- pnpm
- markdownlint

## Node.js

Node.jsはv24を前提にする。

Docusaurus、pnpm、markdownlintなどの実行環境として使う。

## Docusaurus

教材サイト全体の土台として使う。

ホーム、ステップ一覧、ステップ詳細などを静的サイトとして構成する。

ドキュメントのナビゲーション、サイドバー、ページ生成、ビルド、公開まわりをDocusaurusに任せる。

## React

チェックポイントやステップ完了チェックなど、画面上で操作が必要な部分に使う。

ページ全体を複雑なアプリとして作るのではなく、教材ページの一部に必要なUIを追加するために使う。

## TypeScript

ステップ情報、チェックポイント、ユピテル計測値の目安などを型付きで扱うために使う。

カリキュラムが増えても、データ構造の崩れに気づきやすくする。

## Markdown / MDX

教材本文はMarkdownで管理する。

チェックリストや目安表示など、Reactコンポーネントを埋め込みたいページではMDXを使う。

## Corepack / pnpm

パッケージ管理にはpnpmを使う。

pnpmはCorepack経由で利用する。

依存関係のインストール、開発サーバーの起動、ビルド、各種スクリプトの実行はpnpm経由で行う。

Corepackを前提にすることで、プロジェクトで使うpnpmのバージョンを固定しやすくする。

## markdownlint

Markdown文書の見出し構造、リスト表記、空行、ファイル末尾の改行などを確認するために使う。

このプロジェクトではコードよりも教材Markdownの比重が高いため、初期段階のlintはmarkdownlintのみ導入する。

ESLint、Prettier、Biomeなどのコード向けlint / formatterは、ReactやTypeScriptのコード量が増えた段階で必要に応じて検討する。

## データ管理

初期段階では、ステップ情報やチェックポイントはリポジトリ内の静的データとして管理する。

ユーザーごとの練習記録は持たせない。

ステップ完了状態を保存する場合も、まずはブラウザ内のローカル保存で十分かを検討する。

## スタイリング

基本のレイアウトとドキュメント表示はDocusaurusのテーマを使う。

必要に応じて、ステップ一覧、チェックポイント、計測値目安などの表示だけ独自に調整する。

## 公開

静的サイトとして公開できる構成にする。

サーバー処理を前提にしないため、GitHub Pages、Cloudflare Pages、Netlify、Vercelなどの静的ホスティングに載せやすい。

## 初期段階で使わないもの

- バックエンドサーバー
- データベース
- ユーザーアカウント
- ログイン機能
- クラウド同期
- AI解析基盤
- 動画解析ライブラリ
- 計測器との自動連携

## 技術選定の理由

このプロジェクトの中心は、複雑な処理ではなく教材の内容と練習ステップの分かりやすさである。

そのため、最初は静的サイトとして作り、必要な部分だけReactで拡張する構成が適している。

技術的な複雑さを抑えることで、カリキュラム作成とチェックポイント設計に集中できる。
